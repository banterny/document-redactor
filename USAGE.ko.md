## 1. 파일 받기

[최신 릴리즈 페이지](https://github.com/kipeum86/document-redactor/releases/latest)로 가서 **두 파일 모두** 다운로드합니다:

- **`document-redactor.html`** — 도구 본체 (단일 HTML 파일, 약 180 KB)
- **`document-redactor.html.sha256`** — 무결성 검증용 사이드카 파일 (89 바이트)

카카오톡, 이메일, USB로 다른 사람에게서 받은 경우도 괜찮습니다 — 다음 섹션의 검증 단계가 정확히 그 시나리오를 위해 설계되어 있습니다. 보낸 사람을 신뢰할 필요는 없습니다. 해시를 검증하면 됩니다.

## 2. 다운로드 검증

사이드카 파일은 당신이 가진 파일이 공식 배포본과 **바이트 단위로 동일**하다는 것을 수학적으로 증명해줍니다. 공식 릴리즈와 당신의 디스크 사이에서 파일은 다음과 같은 이유로 수정될 수 있습니다:

- 손상된 미러 서버
- 회사의 프록시 혹은 DLP 시스템이 다운로드를 자동 재작성
- 악의적인 네트워크 중간자
- 선의로 보낸 사람이 실수로 재내보내기하거나 재압축

이 모든 시나리오는 SHA-256 해시를 다르게 만듭니다. 검증은 한 줄짜리 명령입니다.

### macOS / Linux

```bash
cd /다운로드받은/경로
sha256sum -c document-redactor.html.sha256
```

기대 출력:

```
document-redactor.html: OK
```

`sha256sum` 이 없는 구형 macOS에서는:

```bash
shasum -a 256 -c document-redactor.html.sha256
```

### Windows (PowerShell)

```powershell
cd C:\다운로드받은\경로

# HTML의 해시 계산
$actual = (Get-FileHash -Algorithm SHA256 document-redactor.html).Hash.ToLower()

# 사이드카에서 기대 해시 읽기
$expected = (Get-Content document-redactor.html.sha256).Split(' ')[0].ToLower()

if ($actual -eq $expected) { "OK" } else { "MISMATCH — 실행 금지" }
```

### Windows (WSL / Git Bash)

macOS / Linux와 동일:

```bash
sha256sum -c document-redactor.html.sha256
```

### `OK` 외 다른 결과가 나온다면

**멈추세요.** HTML 파일을 열지 마세요. 당신이 가진 파일은 공식 배포된 그 파일이 아닙니다. 다음 조치를 해보세요:

1. GitHub 릴리즈 페이지에서 직접 두 파일 모두 다시 다운로드 (다른 사람이 보낸 링크 말고, 브라우저 주소창에 직접 입력)
2. 다른 네트워크에서 재시도 (회사 프록시가 파일을 재작성하고 있을 수 있음)
3. 문제가 계속되면 GitHub 이슈로 제보

## 3. 도구 열기

`document-redactor.html` 을 더블클릭합니다. 기본 브라우저가 `file://` URL로 엽니다. 설치 프로그램 없음, 권한 요청 없음, 네트워크 호출 없음, 툴바 확장 프로그램 없음.

브라우저의 탭 제목은 `document-redactor · offline DOCX redactor` 로 나옵니다. 좌측 상단의 브랜드 영역에 `document-redactor`, 우측 상단에 `0 network requests` 배지 — 말 그대로 0 requests입니다.

### 작동하는 브라우저

- Chrome, Chromium, Brave, Edge (v120+) — 테스트 완료
- Firefox (v120+) — 테스트 완료
- Safari (16+) — 테스트 완료
- ES2022, Web Crypto SubtleCrypto, `file://` origin을 지원하는 모든 최신 브라우저

### 작동하지 않는 브라우저

- Internet Explorer (모든 버전)
- 구형 Safari (<16)
- 파일 API가 제한된 모바일 브라우저 — 기술적으로는 로드되지만 드롭 영역 UX가 폰에서는 어색함

## 4. 첫 마스킹

계약서를 단계별로 마스킹해보겠습니다.

### 4.1 — 파일 드롭

화면 중앙에 "Drop a DOCX file here, or choose a file." 라고 적힌 큰 점선 사각형이 보입니다. 파일 탐색기에서 `.docx` 파일을 이 사각형 위로 드래그하거나, "choose a file" 링크를 클릭해서 선택합니다.

도구는 파일을 메모리로 로드하고, 메모리 안에서 zip 압축을 풀고, 텍스트가 들어있는 모든 scope(본문, 각주, 미주, 댓글, 머리말, 바닥글)를 순회한 뒤 다음 phase로 넘어갑니다. 일반적인 계약서에서는 1초 이내에 완료됩니다.

> **아무것도 당신의 컴퓨터를 떠나지 않습니다.** 파일은 브라우저 탭 안의 JavaScript 변수에 존재합니다. 어떤 서버로도 업로드되지 않고, 디스크에 캐시되지도 않고, 네트워크로 전송되지도 않습니다 — 보낼 서버 자체가 없기 때문입니다.

### 4.2 — 시드 에디터 채우기 (왼쪽 사이드바)

왼쪽 사이드바에 **Seed editor** 가 있습니다. 여기에 당신이 찾아서 마스킹하고 싶은 이름과 문구를 입력합니다.

기본값으로 몇 개의 예시 시드(회사명, 인명)가 들어있습니다. **실제 대상으로 교체하세요.** 한 줄에 하나씩. 예:

```
ABC 법인
XYZ 홀딩스
김철수
Jane Smith
010-1234-5678
```

시드 에디터는 **사용자 지정 개체명**을 위한 곳입니다 — 도구가 알아서 추측할 수 없는 것들(회사명, 특정 인물, 특정 주소 등). 알려진 PII 패턴(전화번호, 이메일, 주민등록번호, 사업자등록번호 등)은 PII 정규식 스윕이 자동으로 감지하므로 여기에 입력할 필요 없습니다.

### 4.3 — 후보 패널 검토 (오른쪽)

파일을 드롭한 뒤, 오른쪽 패널에는 **세 그룹**의 후보가 표시됩니다:

1. **Literal names** — 입력한 각 시드의 모든 변형. `ABC 법인` 을 시드로 입력했다면, `ABC 법인`, `ABC`, 그리고 다른 추출된 하위 문자열들이 각각 행으로 나타납니다.
2. **Defined term labels** — 도구가 계약서의 정의(`"Discloser"란 ~를 말한다` 형식)에서 감지한 역할 용어와 별칭(`the Buyer`, `the Discloser`, `甲`, `매수인` 등). **기본값으로 체크 해제** — [§ 6 D9 정의된 용어 정책](#6-d9--정의된-용어defined-term-정책) 참고.
3. **Auto-detected PII** — 정규식 스윕이 찾은 모든 전화번호, 이메일, 주민등록번호, 사업자등록번호, EIN, 은행 계좌, 신용카드 번호.

각 행에는 체크박스가 있습니다. 그룹 1과 3의 모든 항목은 **기본 체크**, 그룹 2의 항목은 **기본 체크 해제**. 어떤 체크박스든 자유롭게 토글할 수 있습니다.

### 4.4 — Apply 누르기

선택에 만족했으면 패널 하단의 **Apply and verify** 버튼을 클릭하거나, 어디에서든 **⌘/Ctrl + Enter** 를 누릅니다.

도구는:

1. XML을 제자리에서 재작성하여 선택된 모든 후보를 `[REDACTED]`로 치환합니다.
2. 검증 pass 실행 — 출력 바이트를 다시 읽어서 선택된 대상이 0개 남았는지 확인합니다.
3. 출력 파일의 SHA-256 해시를 계산합니다.
4. 원본 대비 단어 수를 체크합니다 (30% 이상 감소하면 의심 플래그).
5. 녹색 배너를 표시합니다: **"Verification passed"** + SHA-256(앞 4자리 + 뒤 4자리) + 원본/결과 단어 수.

### 4.5 — 다운로드

녹색 배너의 **Download** 버튼을 누릅니다. `{원본 파일명}.redacted.docx` 로 저장됩니다. 예를 들어 `NDA_2026_final.docx` 는 `NDA_2026_final.redacted.docx` 가 됩니다.

이 파일이 당신이 전달할 최종본입니다. Word, Google Docs, 혹은 어떤 편집기에서든 열 수 있고, 마스킹된 부분은 `[REDACTED]` 로 인라인 표시됩니다.

## 5. 세 가지 후보 그룹 이해하기

### 그룹 1 — Literal names (빨간 pill)

이 그룹의 모든 항목은 시드 에디터에 입력한 것들과 그 **자동 생성 변형**입니다. `ABC 법인` 을 입력하면 도구가 다음을 제안합니다:

- `ABC 법인` (시드 자체)
- `ABC` (접두 축약)
- 기업명 관용 패턴에 맞는 다른 부분 문자열

**기본 체크 상태** — 명시적으로 마스킹 요청한 대상이기 때문입니다.

**짧은 ROOT 주의.** `ABC` 가 체크된 상태에서 문서에 `abcdefg Co.` 같은 다른 개체명이 있다면, 짧은 root가 그것도 매치합니다. 전체 이름 형태만 마스킹하고 싶다면 `ABC` 를 체크 해제하세요.

### 그룹 2 — Defined term labels (슬레이트 pill)

도구가 다음과 같은 패턴을 보고 감지한 역할 용어와 정의 별칭입니다:

- `"Discloser" means ABC Corporation` → `Discloser`
- `(hereinafter "the Buyer")` → `the Buyer`
- 한국어: `("갑"이라 함은 ABC 법인을 말한다)` → `갑`
- 일본식: `甲`, `乙`

**기본 체크 해제** — 이걸 마스킹하면 `The Buyer acknowledges that the Discloser may disclose…` 가 `The [REDACTED] acknowledges that the [REDACTED] may disclose…` 가 되어 가독성이 사라지고, 다운스트림 AI 도구가 당사자 역할을 이해하지 못합니다. [§ 6](#6-d9--정의된-용어defined-term-정책) 참고.

### 그룹 3 — Auto-detected PII (앰버 pill)

PII 정규식 스윕의 모든 히트:

- 주민등록번호 (`XXXXXX-Xxxxxxx` 형식)
- 사업자등록번호 (`XXX-XX-XXXXX` 형식)
- EIN (미국 사업자 세금 번호, `XX-XXXXXXX` 형식)
- 전화번호 (한국, 국제)
- 이메일
- 한국 은행 계좌 번호
- 신용카드 번호 (Luhn 체크 검증)

**기본 체크 상태**. 각 행에는 감지된 종류가 표시됩니다 (`주민등록번호`, `phone · KR`, `email` 등).

**PII 정규식 스윕은 결정론적입니다.** ML도, 확률 점수도 없습니다. 패턴에 맞으면 나오고, 안 맞으면 안 나옵니다. 패턴은 엄격합니다 — 임의의 13자리 숫자는 "주민번호 같음" 으로 나오지 않고, 실제로 형식에 맞아야만 나옵니다.

## 6. D9 — 정의된 용어(defined term) 정책

일반적인 양당사자 계약서의 정의 섹션은 다음과 같이 쓰여 있습니다:

> This Agreement is made between ABC Corporation ("Discloser") and XYZ Holdings ("Recipient").

이 문장 이후, 계약서의 나머지는 당사자를 전체 이름 대신 "the Discloser"와 "the Recipient"로 지칭합니다. 이때 전체 이름 **그리고** 정의된 용어를 **둘 다** 마스킹하면, 모든 언급이 사라집니다:

> `The [REDACTED] agrees that [REDACTED] may share information with [REDACTED] employees…`

이건 쓸모가 없습니다. 마스킹의 목적은 결국 다운스트림 독자(다른 변호사, AI 도구, NDA 비당사자 등)가 계약서의 구조를 여전히 이해할 수 있게 하는 것입니다. 역할 용어를 유지하면 가독성이 살아있습니다:

> `The Discloser agrees that Recipient may share information with Recipient employees…`

그래서 **정의된 용어는 기본 체크 해제** 입니다.

### 언제 정의된 용어를 켜야 하나

다음 경우에 체크하세요:

1. **같은 역할의 당사자가 3명 이상일 때.** 세 회사가 모두 "Licensee" 라면, `Licensee` 를 마스킹하지 않는 것 자체로는 아무 정보도 숨기지 않습니다 — 그냥 일반 역할 용어이니까요. 하지만 셋 중 두 명이 `Licensee A`, `Licensee B` 라면, 전체 이름만 마스킹하고 `Licensee A` / `Licensee B` 를 남기는 것은 소거법으로 누가 누군지 드러내는 단서가 될 수 있습니다.
2. **결과물이 적대적 독자에게 넘어갈 때.** 받는 쪽이 외부 데이터와 교차 검증해서 재식별을 시도할 가능성이 있다면, 구조 보존보다는 모든 라벨 제거가 더 안전합니다.
3. **공적 기록을 공유할 때.** 당사자명은 가려야 하지만 문서 구조는 온전해야 하는 법원 제출 자료 같은 경우, 정의된 용어는 남기고 독자가 문맥으로 파악하게 두세요.

### 정의된 용어를 꺼두는 경우 (기본값)

일반적인 경우 — 사내 동료에게, 다른 변호사에게, 또는 법률 AI 도구에게 분석을 맡길 때 — 정의된 용어를 남기는 것이 맞습니다. 다운스트림 독자는 계약 구조를 명확히 보고 `[REDACTED]` 해독에 머리를 쓰지 않고 법률 분석에 집중할 수 있습니다.

### 라벨 읽는 법

각 정의된 용어 행에는 **출처 표시**가 아래에 붙어있습니다: `from definition · ABC Corporation`. 어느 개체를 가리키는 라벨인지 보여주므로 항목별로 판단할 수 있습니다.

## 7. 계약서가 아닌 문서

엔진은 텍스트 기반이라 어떤 DOCX에도 동작합니다 — 문서가 계약서든 의견서든 준비서면이든 메모든 내부 노트든 구분하지 않습니다. 계약서가 아닌 경우 사용법:

- **시드 에디터에 대상을 직접 입력하세요.** 판결문에서 무엇을 마스킹할지(사건 번호? 당사자 이름? 판사 이름?) 도구가 추측할 방법이 없습니다. 직접 입력해야 합니다.
- **정의된 용어 감지는 아무것도 찾지 못합니다.** D9 파서는 `"X" means Y` 와 `("Y"이라 함은)` 패턴을 찾는데, 이건 거의 계약서 전용입니다. 계약서가 아닌 문서에서는 정의된 용어 그룹이 비어있는 게 정상입니다.
- **PII 정규식 스윕은 여전히 작동합니다.** 주민번호, 전화번호, 이메일, 사업자번호 — 문서 종류와 무관하게 감지됩니다.
- **Stop-phrase는 계약서 플레이버입니다.** 내부 stop-phrase 리스트(키워드 제안기가 건너뛰는 노이즈 단어들)에는 "제1조", "본 계약", "갑", "을" 같은 계약서 골격 용어가 들어있습니다. 이게 오탐을 일으키지는 않습니다 — detection이 아니라 *제안* 사이드바의 필터링에만 쓰이니까요. 무시해도 됩니다.

**실전 예시 — 판결문 발췌 마스킹:**

1. 시드 에디터:
   ```
   김철수
   박영희
   ABC 법인
   서울중앙지방법원 2024가합12345
   ```
2. PII 스윕이 판결문 텍스트에서 전화번호, 주민번호, 사업자번호를 찾으면 그룹 3에 나옵니다.
3. 정의된 용어 그룹은 비어있습니다 (판결문에 `"X" means Y` 없음).
4. Apply → 검증 → 다운로드.

## 9. 결과물 검증

성공적으로 마스킹이 끝나면 녹색 배너가 결과물의 SHA-256(앞 4자리 + 뒤 4자리)을 보여줍니다. 이 해시를 터미널에서 독립적으로 검증할 수 있습니다:

```bash
sha256sum NDA_2026_final.redacted.docx
```

앞 4자리 + 뒤 4자리가 배너에 표시된 것과 일치해야 합니다. 불일치하면 버그입니다 — 이슈를 남겨주세요.

### 왜 결과물을 해싱하나요?

마스킹한 파일을 다른 사람에게 보냈는데, 받은 사람이 "내가 받은 파일이 네가 보낸 그 파일이 맞는가?" 를 확인해야 할 때, 해시가 계약이 되기 때문입니다. 같은 해시 = 바이트 단위로 동일한 파일. 다른 해시 = 전송 중에 뭔가 변형됨 (이메일 재인코딩, zip 재압축, 회사 DLP 재작성 등).

### 해시는 실행 간에 재현 가능한가요?

**네, 같은 입력 + 같은 선택이면 재현됩니다.** 도구는 출력 zip을 생성하기 전에 모든 zip 엔트리의 내부 타임스탬프를 Unix epoch 0으로 고정합니다. 그 결과:

- 같은 입력과 같은 토글 선택으로 도구를 두 번 돌리면 → 바이트 단위로 동일한 결과물 → 동일한 SHA-256
- 서로 다른 두 사람이 같은 입력과 같은 선택으로 도구를 돌리면 → 동일한 SHA-256

이것이 카카오톡 배포 시나리오를 가능하게 만드는 속성입니다. "내가 보낼 파일의 해시는 이것이다" 라고 말하면, 받은 사람은 받은 파일이 보낸 파일과 같은지 검증할 수 있습니다.

## 10. 문제 해결

### "드롭한 파일이 스피너만 무한정 돌아감"

파싱 단계는 동기 방식이고 빠릅니다 (< 1초). 멈춘 것처럼 보이면:

1. 브라우저 개발자 도구 (F12) → Console 탭. 에러를 찾으세요.
2. 파일이 정말 `.docx` 인지 확인. `.doc` (레거시 바이너리), `.pages`, 기타 비-OOXML 파일은 거부됩니다.
3. 매우 큰 파일(>10 MB)은 몇 초 걸릴 수 있습니다 — 기다려보세요.
4. 위가 모두 아니면: console 에러와 재현 가능한 fixture(실제 파일이 아니라 합성 minimal DOCX)를 첨부해서 버그를 남겨주세요.

### "찾을 줄 알았던 이름이 후보 리스트에 안 나옴"

먼저, 실제로 시드 에디터에 입력했는지 확인하세요. 도구는 다음만 찾습니다:

- 입력한 literal 시드 (와 자동 변형)
- 정의된 용어 패턴 (`"X" means Y` 형식)
- PII 패턴 (정규식 — 전화번호, 이메일, 주민등록번호, 사업자등록번호 등)

특정 회사명을 자동으로 찾을 거라 기대했다면 — v1에는 "고유명사 감지" 가 없습니다 (ML 모델이 필요한데, 이 도구엔 없음). 시드 에디터에 추가하고 파일을 다시 드롭하세요.

### "없는 자리에서 주민번호가 나왔음"

가능성은 낮지만 존재 — 정규식이 엄격하지만 완벽하진 않습니다. 텍스트 내 위치를 확인하세요. 일반적 false positive:

- `2024041012345678` 처럼 우연히 13자리 긴 ID — 정규식은 `[0-9]{6}-[1-4][0-9]{6}` 에 앵커되어 있고 하이픈과 7번째 자리 제약이 있어서 이건 매칭 안 됩니다. 만약 매칭된다면 버그입니다.
- `2024-01-15` 같은 날짜를 `240115` 로 포맷 — 주민번호 패턴과 매치 안 됨.

정말 패턴에 맞는 것 같은 false positive를 보게 되면, 해당 행을 체크 해제한 뒤 Apply 하세요.

### "검증 단계가 실패함"

배너가 빨간색으로: **"Verification FAILED — N occurrences remain"**. 이 경우 마스킹 후 검증기가 출력을 다시 읽어서 여전히 타겟을 발견했다는 뜻입니다. 잘 형성된 DOCX에서는 절대 발생해서는 안 되고 버그를 의미합니다.

우회법: 남아있는 특정 타겟을 체크 해제하고 재실행하세요. 그리고 **버그를 제보**해주세요:
- 재현 가능한 DOCX fixture (또는 minimal 합성)
- 선택한 타겟 목록
- 입력 파일의 SHA-256

### "도구가 느림"

파싱 ~200ms. 감지 ~300ms (일반 계약서). Apply ~500ms + write-back. 총 2초 이내 end-to-end.

정말 느리다면 (>10초), 가장 흔한 원인은 수천 paragraph가 있는 매우 큰 DOCX입니다. 개발자 도구 Performance 탭으로 프로파일링할 수 있습니다. Fixture 크기와 함께 버그를 제보해주세요.

### "50 MB 파일을 드롭했더니 브라우저가 멈춤"

v1에는 명시적 크기 cap이 없지만, 브라우저는 수백 MB 파일을 메모리에 전부 올리는 걸 힘들어합니다. 일반적인 법률 문서는 <5 MB입니다. 파일이 유난히 크다면(스캔 PDF를 Word로 변환하면서 이미지가 대거 들어간 경우 등), [§ 11](#11-이-도구가-하지-않는-것들)의 OCR / 이미지 주의사항이 적용됩니다 — 이미지 속 텍스트는 어차피 마스킹 못 하니까 이미지가 많은 파일에는 이 도구의 이점이 없습니다.

### "모바일 버전이 이상함"

720 px 미만에서는 3컬럼 레이아웃이 2컬럼으로 축소됩니다. 400 px 미만에서는 요소들이 겹치기 시작합니다. 이 도구는 v1에서 데스크톱 전용으로 의도되어 있습니다. 모바일 polish는 v1.x 로드맵에 없습니다.

### "마스킹을 되돌리고 싶음"

되돌릴 수 없습니다 — 결과물 파일에는 `[REDACTED]` 가 이미 들어가 있습니다. 하지만 다음은 가능합니다:

1. 드롭 영역으로 돌아가기
2. **원본** 파일을 다시 드롭 (도구는 파싱 사이 상태를 저장하지 않음)
3. 이제 유지하고 싶은 타겟을 체크 해제
4. Apply 재실행

도구는 디스크 위의 원본 입력 파일을 절대 수정하지 않고, 읽기만 합니다.

## 11. 이 도구가 하지 않는 것들

v1 한계입니다. 대부분 v1.x에서 계획되어 있습니다.

- **PDF 입력 불가 — 향후 업데이트 예정.** v1은 `.docx` 전용. PDF를 먼저 Word로 변환하거나 PDF 도구의 내보내기 기능을 써서 DOCX로 만든 뒤 이 도구에 넣으세요.

- **OCR 없음.** DOCX에 텍스트 이미지가 포함되어 있다면(스캔본, 다른 문서의 스크린샷) 그 이미지 속 텍스트는 보이지 않습니다. 미리 이미지 편집기에서 마스킹하거나, 별도 OCR 파이프라인을 쓰세요.

- **손글씨 서명 이미지 없음.** OCR과 동일 — 픽셀입니다, 텍스트가 아님.

- **SmartArt 또는 WordArt 텍스트 없음.** 특수 OOXML 구조라 v1의 scope walker가 순회하지 않습니다. 사용했다면 내보내기 후 수동 확인하세요.

- **임베디드 Excel / PowerPoint 객체 없음.** OLE 임베디드 객체는 opaque blob으로 취급됩니다. 네이티브 DOCX 표(일반 Word 표)는 **완전히 처리됨**.

- **폼 필드 또는 콘텐츠 컨트롤 없음.** Microsoft Word는 폼 필드에 structured document tag(`<w:sdt>`)를 지원합니다. v1은 이것들을 텍스트 run 목적상 텍스트로 취급하지만 필드 의미 구조는 이해하지 못합니다.

- **매크로 또는 VBA 없음.** `.docm` 파일(매크로 활성화 Word)은 v1에서 지원하지 않습니다. 먼저 `.docx` 로 변환하세요.

- **프리뷰 패널 클릭-선택 없음.** 프리뷰는 로드된 파일 이름과 자리표시자만 보여주고, 후보 검토는 전적으로 오른쪽 패널에서 일어납니다.

- **도구 내부에 undo / redo 없음.** 각 "Apply" 는 일회성입니다. 마음이 바뀌면 드롭 영역으로 돌아가 원본 파일을 다시 드롭하세요.

- **세션 간 상태 영속화 없음.** 도구는 아무것도 기억하지 않습니다. 탭을 닫고 다시 열면 처음부터 시작입니다.

- **배치 처리 없음.** 한 번에 파일 하나. 대량 마스킹은 파일마다 한 번씩 실행.

- **정책 파일 / 팀 공유 없음.** 모든 사용자는 자신의 시드 목록을 독립적으로 유지합니다. v1에는 시드 import/export 없음.

## 12. 프라이버시 정책

**우리는 아무것도 수집하지 않습니다. "우리" 가 없기 때문입니다.**

`document-redactor` 는 HTML 파일입니다. 당신의 컴퓨터에서, 당신의 브라우저 탭 안에서, 당신의 디스크에서 실행됩니다. 백엔드 없음. 당신의 데이터가 기록될 수 있는 데이터베이스 없음. 애널리틱스 엔드포인트 없음. 에러 리포팅 서비스 없음. "사용 텔레메트리" 없음. 기능 플래그 서비스 없음. 외부 세계와 연결할 수 있는 경로 자체가 없습니다:

1. 소스 코드에 `fetch`, `XMLHttpRequest`, `WebSocket`, 기타 네트워크 API 호출이 없음.
2. 빌드된 HTML이 빌드 타임에 위 토큰이 0개임을 확인됨.
3. 런타임 Content-Security-Policy 가 그런 호출이 있어도 차단.
4. 위 세 가지 모두 직접 검증 가능.

이 도구로 파일을 마스킹할 때, 당신이 드롭한 파일, 당신이 입력한 시드 목록, 당신이 토글한 선택 상태, 당신이 다운로드한 결과 파일 — 이 모두는 브라우저 탭의 메모리와 디스크에만 존재합니다. 탭을 닫으면 모든 것이 사라집니다.

이 설명과 모순되는 동작을 발견하신다면, 그것은 버그이고, 이 프로젝트에서 가장 중요한 버그입니다. 즉시 이슈를 남겨주세요.
